# baseproof v1.37.0 SDK Adoption Runbook

Operator-facing rollout, configuration, and rollback protocol for the ledger's adoption of baseproof SDK v1.37.0. This release makes the admission gate polymorphic at the cryptographic seam: the ledger admits every algorithm + DID method the SDK can verify.

## TL;DR for operators

1. **Bump the binary; do nothing else.** Default v1.37.0 behaviour is identical to v1.36.0 for ECDSA entries, plus automatic admission of Ed25519 / ML-DSA / SLH-DSA entries from any did:key or did:web actor. No env-var changes required.

2. **Want to admit smart-contract-wallet litigants (Safe, Argent, Coinbase Smart Wallet)?** Set the seven `LEDGER_EIP1271_*` env vars per the Tier 2 procedure below.

3. **Want to admit post-quantum signatures?** Nothing to do. The polymorphic path handles ML-DSA / SLH-DSA automatically.

4. **Rollback?** Binary revert to v1.36.0. The polymorphic path produces wire-compatible entries; no on-log state needs reverting.

## Scope of this release

v1.37.0 is the largest admission-surface change since v1.32.0, but at the wire-format level it is strictly additive. Pre-v1.37 entries continue to verify; v1.37+ entries that use new algorithms are admitted by the polymorphic verifier registry. Three tiers ship together:

- **Tier 1**: Polymorphic admission seam — `*did.VerifierRegistry` wired into the admission path
- **Tier 2**: EIP-1271 smart-contract-wallet verification — feature-flagged, OFF by default
- **Tier 3**: Post-quantum (ML-DSA-65, ML-DSA-87, SLH-DSA-128s) — automatic via Tier 1

## Tier 1 deploy (the default)

### Pre-deploy verification

```bash
# Confirm the SDK pin
grep "clearcompass-ai/baseproof " go.mod
# Expected: github.com/baseproof/baseproof v1.37.0

# Confirm wire.go composes the polymorphic chain
grep -n "buildIdentityDeps\|MethodRouter\|CachingResolver" cmd/ledger/boot/wire/wire.go

# Confirm the polymorphic admission entry exists
grep -n "VerifyEntryAllSignaturesWithVerifier" admission/multisig_verifier.go

# Confirm full test suite green
go test ./... 2>&1 | grep -E "^FAIL"
# Expected: no output
```

### Deploy

Binary swap. The wire phase constructs the `*did.VerifierRegistry` at boot. Look for this log line on startup:

```
INFO v1.37.0: polymorphic verifier registry wired methods=[did:key did:web did:pkh] eip1271_enabled=false cache_ttl=5m
```

If you do not see this line — or if `eip1271_enabled` is reported as `true` when you expected `false` — investigate before promoting the binary further.

### Post-deploy verification

1. **ECDSA backward compatibility.** Submit a normal did:key + ECDSA entry. It must admit identically to v1.36.0 — same SCT, same signature receipt shape.

2. **Ed25519 admits.** Submit a did:key + Ed25519 entry. Pre-v1.37.0 this would have failed at admission with `ErrUnsupportedSignatureAlgo` (HTTP 422). At v1.37.0 it must succeed with HTTP 202.

3. **PQ admits.** Submit a did:key + ML-DSA-65 entry. Same expectation as Ed25519. The test fixture in `admission/multisig_verifier_polymorphic_test.go::TestVerifyEntryAllSignaturesWithVerifier_DIDKey_MLDSA65` is the canonical reference.

## Tier 2 enable — EIP-1271 (smart-contract wallets)

### Pre-conditions

- You have at least two **independent** Ethereum JSON-RPC endpoints. The SDK enforces K-of-N with K≥2 by design (Byzantine-rogue protection against a single rogue provider). Use different vendors (Alchemy + Infura + QuickNode) or your own self-hosted node + one commercial provider.

- You have a **recent finalized block** to pin verification to. v1.37.0 first-ship uses `StaticBlockProvider`. Operators supply the block at deploy time and refresh via env var + restart cadence. A reasonable rollout target is "head − 64 blocks" (~13 minutes of finality margin on Ethereum mainnet, comfortably past every reorg in practice).

### Env-var contract

| Variable | Required when | Format | Example |
|---|---|---|---|
| `LEDGER_EIP1271_ENABLED` | Always (to enable) | `true` / `false` | `true` |
| `LEDGER_EIP1271_EXECUTORS` | Enabled | `id1=url1,id2=url2[,...]` | `alchemy=https://eth-mainnet.alchemyapi.io/v2/KEY,infura=https://mainnet.infura.io/v3/KEY` |
| `LEDGER_EIP1271_QUORUM_K` | Optional | uint ≥ 2 | `2` (default) |
| `LEDGER_EIP1271_CHAIN_ID` | Enabled | uint (EIP-155) | `1` (Ethereum mainnet) |
| `LEDGER_EIP1271_BLOCK_NUMBER` | Enabled | uint | `19000000` |
| `LEDGER_EIP1271_BLOCK_HASH` | Enabled | 32-byte hex (optional `0x` prefix) | `0x71f...` |

Misconfiguration is a fail-fast startup error, not a silent degrade to disabled. The ledger does NOT start with `LEDGER_EIP1271_ENABLED=true` and a single executor; it aborts with `ErrEIP1271RequiresExecutors`. This is intentional — silent EIP-1271 disabled-mode would let an operator believe Safe-wallet litigants were being admitted when they were actually being rejected.

### Cost expectations

| Scenario | TPS | EIP-1271 share | Monthly Ethereum spend |
|---|---|---|---|
| Default | — | disabled | **$0** |
| Launch (low) | 100 avg | 5% | ~$50 |
| Mature | 200 avg | 5% | ~$100 |
| Peak (sustained) | 1000 | 5% | ~$300 |
| Pathological | 1000 sustained | 10% | $1-2K (custom-tier) |

Costs assume Alchemy / Infura per-CU billing. **Self-hosted Ethereum archive node** runs ~$200-500/month flat; above ~$500/month in per-CU spend, switch to self-hosting.

### Refresh cadence for the block pin

`StaticBlockProvider` returns the same `(BlockNumber, BlockHash)` for every verification. As the chain advances, this becomes increasingly stale. EIP-1271 isValidSignature lookups remain valid against the pinned block because Safe / Argent / Coinbase Smart Wallet contracts are append-only — owners can be added at later blocks but never removed at the pinned block.

Recommended refresh cadence: **24 hours**. Update the env vars + binary restart. A future SDK release will ship a dynamic head-pinning provider that obviates the need for this cadence.

### Post-enable verification

1. **Boot log.** Look for:
   ```
   INFO eip-1271 verification enabled executors=2 quorum_k=2 chain_id=1
   INFO v1.37.0: polymorphic verifier registry wired methods=[did:key did:web did:pkh] eip1271_enabled=true cache_ttl=5m
   ```

2. **Test submission from a Safe wallet** in a staging environment. The wallet owner signs an EIP-712 typed message off-chain; the relayer submits the entry with `Signature.AlgoID = SigAlgoEIP1271`. Expected outcome: HTTP 202 with a populated `Web3VerificationReceipt` (executor IDs + ResultHash + block tag).

3. **Watch the metrics.** Cost overrun protection comes from the `EIP1271BatchCache` hit rate. A persistent low hit rate (< 50%) at scale indicates either unusual wallet diversity or a cache misconfiguration. Investigate before the next monthly billing window.

## Tier 3 — Post-quantum

No operator action required. The polymorphic admission path handles ML-DSA-65, ML-DSA-87, SLH-DSA-128s automatically.

**Operational considerations for PQ at scale:**

- Public key sizes: ML-DSA-65 is 1952 bytes, ML-DSA-87 is 2592 bytes, SLH-DSA-128s is 32 bytes. Signature sizes are correspondingly large for ML-DSA (3K-4K bytes) and very large for SLH-DSA (~7K bytes). Entry sizes grow accordingly; budget for higher bytes-per-entry if PQ adoption is non-trivial in your network.

- Verification CPU cost: ML-DSA-65 verifies in ~1ms (similar to ECDSA). SLH-DSA-128s verifies in ~5-10ms (slow signer, fast verifier; still fine at 1000 TPS).

- did:key encoding for PQ uses draft-status multicodec prefixes — see the SDK's [v1.37.0 release notes](https://github.com/baseproof/baseproof/releases/tag/v1.37.0). Until the multicodec assignments are marked permanent, gate PQ did:key acceptance behind a per-network feature flag if cross-implementation interop matters. did:web is unaffected (it uses W3C type strings, not multicodec).

## Counter reasons (admission rejection telemetry)

Per-rejection counters are unchanged from v1.36.0. The new failure modes from the polymorphic path surface through the same counters:

| Reason | When it fires |
|---|---|
| `signature_invalid` | SDK verifier returned a per-algorithm error (tampered sig, wrong key, etc.) |
| `signer_did_resolution` | did:web fetch failed, did:key parsed wrong, did:pkh address malformed |
| `unsupported_signature_algo` | SDK doesn't have a verifier for the algoID (very rare post-v1.37.0; possible for new algos in future SDK releases the ledger hasn't yet adopted) |

## Rollback procedure

The release is fully wire-compatible with v1.36.0:

- No on-log entry kinds were introduced or changed
- No counter names were changed
- No endpoint paths were changed
- Entries admitted at v1.37.0 using ECDSA are byte-identical to v1.36.0
- Entries admitted at v1.37.0 using Ed25519 / EIP-191 / EIP-712 / EIP-1271 / ML-DSA / SLH-DSA would be **rejected** at v1.36.0 (the old ECDSA-only guard)

To roll back, redeploy the v1.36.0 binary. No on-log state needs reverting; no operator config needs adjusting. **Consequence**: new Ed25519 / PQ / smart-contract-wallet submissions will start rejecting with `ErrUnsupportedSignatureAlgo` until the v1.37.0 binary returns. Submissions admitted during the v1.37.0 window remain valid on-log forever — the SDK verifier on every consumer still recognizes them.

## Acceptance criteria

Per network, v1.37.0 is "fully adopted" when:

1. The v1.37.0 binary is running on every node.
2. `go test ./...` is green on the running ledger source.
3. The "v1.37.0: polymorphic verifier registry wired" log line appears at boot.
4. At least one Ed25519-signed test entry has been admitted (proves Tier 1 unlock is active).
5. (Optional, per operator decision) Tier 2 enabled: at least one EIP-1271 Safe-wallet entry admitted from a test wallet.

When 1-4 hold for 7 consecutive days, v1.37.0 is operationally adopted for that network.
