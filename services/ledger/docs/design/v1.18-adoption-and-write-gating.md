## 0. Topology (verified)

| Layer | Role | Evidence |
|---|---|---|
| **JN exchange** | Preventative **gate** ("is this valid?"). build → sign → submit; `SubmitGater.Admit` runs before bytes leave the exchange. | `judicial-network/api/exchange/handlers/entries.go:10-15`; gate at `handlers/submit_gate.go` |
| **Ledger** | Dumb **sequencer** + **payment** ("can you pay?") + **serving**. Structural/cryptographic admission only. | `ledger/api/submission.go`; `admission/feature_flags.go` |
| **Auditor** | **Detective**. Post-hoc as-of re-verification. | `tooling/services/auditor/…` |

**No-duplication invariant (structural):** the SDK authority engine is consumed
in exactly two places — the **JN SubmitGate** (preventative) and the **auditor**
(detective) — and **nowhere in the ledger**.

Corrections to earlier drafts, proven by code:
- The ledger's admission **policy gate was default-`ON`**, not OFF
  (`feature_flags.go:135` was `Policy: envFlagWithDefault(..., true)`, flipped by
  PR-I). This is now moot — it has been removed (A1).
- **The gate was advisory, not enforced.** The ledger accepted any
  structurally-valid entry via Bearer-session *or* PoW stamp
  (`submission.go:622-659`, `middleware/auth.go:78-104`) and verified **no gate
  identity**; the Bearer DID was used only to deduct a credit
  (`submission.go:804-814`). A token-holder (or stamp-solver) could write
  **directly, around the gate**. This is the zero-trust hole the write-path
  section closes.
- **The gate attests nothing today.** `BundleSubmitGate` holds **no signing
  key**; its `CosignatureVerdict` is transient. Authorization is **self-asserted**
  — `PayloadRoleResolver` reads the submitter's own `signed_by_capacities` with
  **no chain walk** (`verification/payload_role_resolver.go:58-82`).

## 1. The two axes (the core structural rule)

A submission is judged on **two orthogonal axes**, by **independent** ledger
admission stages that never consult each other:

| Axis | Question | Owner | Cryptographic form |
|---|---|---|---|
| **Gating** | **"Is this valid?"** (authorization) | JN SubmitGate (preventative) + auditor (detective) — run the SDK engine | a **detached Web3-EOA authorization signature** from an on-log admission authority |
| **Payment** | **"Can you pay for this submission?"** | the ledger (anti-spam / billing) | **Mode A** (session→credit) *or* **Mode B** (PoW stamp) — unchanged |

The separation is a code-structural rule: gating reads only the authorization
signature + the on-log admission keyset; payment reads only credit/stamp.
Neither can weaken the other; either can be replaced without touching the other.
They must never be **confused**: payment is not authorization, and authorization
is not payment.

**Controlled co-mixing (explicit, not implicit):** a per-log policy *may* let a
valid gating signature **also satisfy** the payment axis (a gate-vetted write is
inherently rate-limited → skip the PoW stamp). This is a deliberate, configured
bridge — the gate signature counts as "paid" — but the two stages stay distinct
in code.

## 2. Write-path gating (the zero-trust fix)

### 2.1 Identity: Web3 EOA, on evidence

The SDK signature registry is **Web3-first** (`core/envelope/signature_algo.go`):
`SigAlgoECDSA` (secp256k1 raw, 0x0001), `SigAlgoEIP191` (0x0003), `SigAlgoEIP712`
(0x0004), `SigAlgoEIP1271` (smart-contract wallet, 0x0006), `SigAlgoJWZ`
(0x0005), and `SigAlgoEd25519` (0x0002) — the last deliberately avoided for named
actors (the ledger's own note: *"an Ed25519-signed entry cannot be verified by
an EVM contract — no Ed25519 precompile"*).

The scalability cliff is EIP-1271: verification is an **`eth_call`** to the
wallet contract, and the cache is deliberately **batch-scoped + block-pinned**,
never global (`did/eip1271_cache.go` — a global TTL cache is a temporal-replay
hole when an SCW revokes a key).

**Verdict for the admission key — scalable AND verifiable:**

| Scheme | Verify cost | EVM-verifiable | Per-write @ 1K+ TPS |
|---|---|---|---|
| **EOA secp256k1** (ECDSA / EIP-191 / EIP-712) | local `ecrecover`, ~tens of µs | yes | **yes** |
| EIP-1271 (contract) | `eth_call` RPC (batch-cached, block-pinned) | yes | no — RPC bottleneck |
| Ed25519 | local, cheap | **no** (no EVM precompile) | breaks the uniform Web3 model |

⇒ The admission authority is a **Web3 EOA** (secp256k1), exactly like other named
actors. Multi-party control = a **K-of-N set of EOAs** (still all local
`ecrecover`), never one EIP-1271 contract on the hot path. **No new crypto
primitive** — the gate's approval is verified by the *same* Web3 signature path
that verifies named-actor entry signatures. The gate simply **is** a named actor
(an "admission authority").

### 2.2 Mechanism (ephemeral, non-polluting)

1. On `Admit` success the JN gate (a Web3-EOA admission authority) produces a
   **detached** signature over `H(canonical(entry))`, **bound to the entry's
   as-of anchor** — the embedded cosigned head / referenced root the entry's
   authority is pinned to. (Entries already carry an embedded cosigned tree head
   verified at `submission.go:534-552`.)
2. It rides to the ledger **out-of-band** (HTTP header), alongside the
   **unmodified** entry bytes.
3. The ledger verifies the signature — `ecrecover` against the **as-of on-log
   admission keyset** (an `admission_authority` projection, rotatable, resolved
   with the same as-of pattern as `[]RotationRecord`) — as a **separate admission
   stage**, then **drops it**. Never sequenced, never stored, zero entry-byte
   growth across 10B.
4. Replay is harmless: the ledger already **dedups by canonical hash**
   (`submission.go:676-689`), so a captured signature can only re-admit the
   identical, idempotent entry.

### 2.3 Surviving the two-part submission

The ledger's submission is two-part: **Part 1** synchronous — admission →
**payment** (`deductCreditModeA`, `submission.go:717`, before WAL) → `WAL.Submit`
(durable) → **SCT promise** (202); **Part 2** asynchronous — Tessera assigns the
sequence number, the builder projects to the SMT.

Gating survives because:
1. The authorization binds **content + as-of anchor**, both **fixed at
   build/sign** — before Part 1 and Part 2. Part 2 assigns a position, but
   authority is pinned to the **anchor**, not the position.
2. Gating is verified in **Part 1, before the SCT promise** — the ledger never
   *promises* to include an unauthorized entry (mirrors how payment already gates
   the SCT, `submission.go:745-747`).
3. **Part 3 (auditor)** re-derives authority **as-of the same anchor** from the
   final sequenced log. The ephemeral signature is the real-time chokepoint; the
   log is the durable proof. We never store — and never trust — "the gate said
   so"; we re-prove it.

## How we decide **de-pollution GA** (the ledger cutover — a ledger milestone, NOT an SDK tag)
SDK v1.18 shipped the tile core + decision harness; v1.19.0 the gating/credit
`authz`; v1.20.0 the admission policy + bootstrap genesis. **De-pollution GA = the
ledger cutover** (incremental SMT-tile emission as default + full PG de-pollution:
builder reads from tiles, drop `jellyfish_nodes`). *(Named "v1.20.0" in an earlier
draft — corrected here to avoid colliding with the SDK v1.20.0 release.)*
Flip only when **all four** hold:
1. **SDK correctness (green):** `BuildDirtyTiles ≡ BuildTiles` property test
   (byte-identical proofs + oracle coverage).
2. **Ledger shadow:** A4 with the full-tiler as default + `BuildDirtyTiles` in
   shadow; incremental set serves byte-identical proofs under real load.
3. **Scale:** sustained 1K TPS + 5K spike at 1B then 10B node counts — p99 commit,
   builder lag within MMD, tile cache hit-rate, CDN offload, measured `jellyfish_nodes`
   row reduction.
4. **Operational:** JN (Part C) + auditor (Part B) run end-to-end against served
   tiles; cross-log/as-of verdicts reproduce.

Until then v1.18 serves via the full-tiler / background regeneration (correct, just
not maximally efficient) and PG keeps the hot node cache.
