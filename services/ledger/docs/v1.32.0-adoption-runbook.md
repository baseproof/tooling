# v1.32.0 SDK Adoption Runbook

**Status:** Active. Applies to clearcompass-ai/ledger after PR #173 merges.
**Audience:** Network operators rolling out baseproof v1.32.0 across the 15-network footprint.
**Scope:** L1–L5 backdoor closures, env-var deprecation, fail-closed traps, observability signals.

---

## What v1.32.0 changes (one paragraph)

Five backdoors closed in the ledger. Each one had the same shape: a load-bearing identity or URL the ledger trusted from operator config (env var, deployment manifest) — an attacker who could edit that config could silently redirect cosignature traffic, accept fabricated audit claims, or replace upstream-anchoring parent URLs without producing any on-log evidence. v1.32.0 moves the authoritative source for each of these to on-log entries (`WitnessEndpointDeclarationV1`, `WitnessIdentityLabelV1`, `AuditorRegistrationV1`) that are network-signed, cosigned-tree-head-anchored, and walked deterministically by every consumer. The config-driven paths remain as **canary fallbacks** so the bootstrap window doesn't strand the ledger — operators see them disappear from the observability signal as each network completes its cutover.

| Layer | What was config-trusted | v1.32.0 authoritative source | Backdoor closed by |
|-------|-------------------------|------------------------------|--------------------|
| L1 | `LEDGER_WITNESS_ENDPOINTS` | `WitnessEndpointDeclarationV1` walker via `*discover.DefaultAuthoritativeResolver` | `witnessclient/head_sync.go` snapshot at construction |
| L2 | (no gate at all — accepted any signed finding) | `AuditorRegistrationV1` walker + `AuthorizedFor(kind)` check | `gossipnet/auditor_scope_gate.go` decorator on `sdkgossip.Store.Append` |
| L3 | (no surface) | Three new HTTP endpoints for the materialized projection | `api/labels.go`, `api/auditors.go`, `api/witness_endpoints.go` |
| L4 | (no per-kind validation) | SDK's per-payload `Validate()` at admission | `admission/network_payload_validator.go` step 4g |
| L5 | `LEDGER_PARENT_ADMISSION_URL` | `FederationGraph` walker via `*DefaultAuthoritativeResolver.ResolvePeer` | `anchor/resolved_submit.go` per-publish resolve |

## Rollout order (per network)

A network's rollout is one of two flows depending on whether the network already has an admission authority publishing `WitnessEndpointDeclarationV1` and `AuditorRegistrationV1` entries on its log.

### Flow A — admission authority already publishing on-log

1. **Pre-flight (no traffic shift)**
   - Deploy the v1.32.0 ledger binary with both old env vars AND on-log walker active.
   - Verify `endpoint_source{surface="witness", source="on_log_resolver"}` is incrementing for every ledger restart.
   - Verify `endpoint_source{surface="parent", source="on_log_resolver"}` is incrementing per anchor publish (one per Part II.9 tick).
   - Verify `auditor_scope_reject_total{reason="not_registered"}` is **zero** (if non-zero, a known auditor is missing from the on-log registry — fix before proceeding).

2. **Drop the canary env vars**
   - Remove `LEDGER_WITNESS_ENDPOINTS` from the deployment manifest.
   - Remove `LEDGER_PARENT_ADMISSION_URL` from the deployment manifest.
   - Redeploy.

3. **Verify cutover complete**
   - `endpoint_source{source="config_canary_fallback"}` MUST stay at zero.
   - `endpoint_source{source="on_log_resolver"}` MUST keep incrementing.
   - `auditor_scope_reject_total{reason="no_registry"}` MUST stay at zero (a non-zero would mean the gate is fail-closing because the registry source isn't wired).

### Flow B — admission authority does NOT yet publish on-log

1. **Stay on canary** (no operational change). Deploy v1.32.0 with both env vars set. The ledger will:
   - Log at boot: `witness/head_sync: endpoint source source=config_canary_fallback ...`
   - Record `endpoint_source{source="config_canary_fallback"}` per snapshot/publish.
   - Reject every gossip finding event with `gossipnet.auditor_scope.reject reason="no auditors registered on this log"` if no `AuditorRegistrationV1` entries exist.

2. **Mint the on-log entries** (one-time per network)
   - `WitnessEndpointDeclarationV1` per witness in the network's set. Until this lands, L1 stays on canary.
   - `WitnessIdentityLabelV1` per witness PubKeyID. Optional but recommended for operator dashboards.
   - `AuditorRegistrationV1` per authorized auditor with the correct `Scope` mask. Until this lands, L2 rejects every finding event.
   - `FederationGraph` parent entries. Until this lands, L5 stays on the parent canary.

3. **Verify each entry was admitted**
   - Each entry MUST be admitted under the network's existing `SignaturePolicy` + `AttestationPolicy` (no new gate code; the entries are admitted exactly like any other signed entry).
   - Each entry MUST be cosigned (visible at a `tree_size` after admission).
   - L4 (`admission/network_payload_validator.go`) runs the SDK's per-payload `Validate()` at the front door — a 422 at admission means the on-log entry was structurally malformed; the operator publishing it must fix the field the error names.

4. **Proceed to Flow A.**

## Environment-variable deprecation timeline

| Env var | v1.32.0 role | Deprecation target |
|---------|--------------|--------------------|
| `LEDGER_WITNESS_ENDPOINTS` | _(was: canary fallback for L1)_ | **Removed — PRE-11 Phase B (#114).** Witness dial URLs now resolve only from on-log `WitnessEndpointDeclaration` records; there is no witness config dial-list and `surface="witness"` never emits `config_canary_fallback`. |
| `LEDGER_PARENT_ADMISSION_URL` | Canary fallback for L5 per publish | Drop after `endpoint_source{surface="parent", source="config_canary_fallback"}` is zero for 7 days |
| `LEDGER_WITNESS_ENDPOINT_SCHEMA` | _(resolver locator)_ | **Removed — PRE-11 Phase B (#114)** |
| `LEDGER_WITNESS_LABEL_SCHEMA` | _(resolver locator)_ | **Removed — PRE-11 Phase B (#114)** |
| `LEDGER_AUDITOR_REGISTRATION_SCHEMA` | _(resolver locator)_ | **Removed — PRE-11 Phase B (#114)** |

PRE-11 Phase B (#114) **retracts the prior "permanent" note.** These five schema-position env vars (the three above plus `LEDGER_AUDITOR_SCOPE_AMENDMENT_SCHEMA` and `LEDGER_ANCHOR_TARGET_SCHEMA`) were the `AuthoritativeResolver`'s record *locators*, not admission inputs. The resolver now locates every platform-family declaration **by-kind** from the `idx_entry_kind` index (Phase A, #117), default-ON — that index *is* their on-log replacement, so the resolver is never dormant behind operator config.

## Fail-closed traps (read this before deploying)

### Trap 1 — auditor registry empty but gate wired

Symptom: every finding event rejected with `reason="no auditors registered on this log"`.

Root cause: `cfg.AuditorRegistrySource` is non-nil (gate is active), but no `AuditorRegistrationV1` entries have been admitted yet.

Fix: admit at least one `AuditorRegistrationV1` entry with a non-zero `Scope` mask covering the kinds the network wants to accept. Until this entry lands, the gate is fail-closed by design.

Watch metric: `auditor_scope_reject_total{reason="no_registry"}`. Expect non-zero during bootstrap window, then zero.

### Trap 2 — `WitnessEndpointDeclarationV1` rejected with `ErrSchemaInvalid`

Symptom: operator publishes a `WitnessEndpointDeclarationV1` entry, admission rejects with `ErrSchemaInvalid`.

Root cause: the schema-ID in the entry's header doesn't match the network's **published schema registry** for that kind. (Historical note: pre-PRE-11 this trap blamed `LEDGER_WITNESS_ENDPOINT_SCHEMA` — but that env var was the resolver's locator, never an admission input, and is removed in PRE-11 Phase B (#114).)

Fix: confirm the entry's schema-ID matches the network's published schema registry. The L4 validator runs the SDK's structural `Validate()` and surfaces the field-level error verbatim, so the 422 body identifies the broken field.

### Trap 3 — auditor `Scope` mask too narrow

Symptom: a registered auditor publishes a finding the operator EXPECTED to accept, the gate rejects with `reason="registered but scope does not cover event Kind"`.

Root cause: the on-log `AuditorRegistrationV1` entry's `Scope` field was set without the required bit.

Fix: publish a **superseding** `AuditorRegistrationV1` entry for the same `AuditorDID` with the correct expanded `Scope`. The walker resolves to the most-recent record for the DID at the `asOf` position; the new record takes effect at admission time.

Watch metric: `auditor_scope_reject_total{reason="scope_mismatch"}`. High-priority alert.

### Trap 4 — L5 fallback URL ALSO points at attacker

The L5 closure is meaningful only if `LEDGER_PARENT_ADMISSION_URL` is correctly set during the bootstrap window. If the env var was tampered before v1.32.0 deployed, the canary path is still poisoned. Audit: confirm `LEDGER_PARENT_ADMISSION_URL` matches the federation parent's published `FederationGraph` URL **at deploy time**, then move to on-log via Flow A.

### Trap 5 — resolver works but URL is wrong

Symptom: `endpoint_source{source="on_log_resolver"}` is incrementing, but cosignature requests fail with TLS hostname mismatch.

Root cause: the on-log `WitnessEndpointDeclarationV1` entry has a URL that doesn't match the witness's served certificate.

Fix: publish a superseding `WitnessEndpointDeclarationV1` for the affected PubKeyID with the correct URL. The walker resolves to the most-recent record at the next ledger restart (L1 is once-per-construction).

## Observability signals (Tier E)

Two OTel `Int64Counter` instruments are bound to the global meter in `observability/v1_32_counters.go`.

### `endpoint_source{source, surface}`

Recorded by L1 (`witnessclient/head_sync.go`) once per HeadSync construction, and by L5 (`anchor/resolved_submit.go`) once per parent publish.

Labels:
- `source`: `on_log_resolver` | `config_canary_fallback` | `none`
- `surface`: `witness` (L1) | `parent` (L5)

Alerts:
- **page** if `endpoint_source{source="none"}` increments — both resolver and canary missed, publish failing.
- **info** when `endpoint_source{surface="X", source="config_canary_fallback"}` hits zero across a peer set — that's the rollout-complete signal.

### `auditor_scope_reject_total{reason, kind}`

Recorded by L2 (`gossipnet/auditor_scope_gate.go`) on every fail-closed reject.

Labels:
- `reason`: `no_registry` | `registry_error` | `not_registered` | `retired` | `scope_mismatch`
- `kind`: SDK gossip Kind string (`BP-GOSSIP-EQUIV-V1`, `BP-GOSSIP-SMT-REPLAY-V1`, `BP-GOSSIP-HIST-REWRITE-V1`)

Alerts:
- **page** on any `reason="scope_mismatch"` — an auditor exceeded its scope. Either a deployment running stale scope, or genuine attempted unauthorized publish.
- **page** on sustained `reason="not_registered"` from a known-auditor DID — config drift between auditors and the on-log registry.
- **info** on `reason="no_registry"` during bootstrap — expected until the registry is populated.

## Rollback procedure

The v1.32.0 ledger is backward-compatible with a v1.31.0 admission authority during the bootstrap window — that is the design intent of the canary fallbacks. Rollback steps:

1. **Re-set the env vars** (if previously dropped): `LEDGER_WITNESS_ENDPOINTS`, `LEDGER_PARENT_ADMISSION_URL`. The canary paths are still wired in v1.32.0 code; restoring the env vars restores the pre-v1.32.0 behaviour for L1 and L5.

2. **Disable the L2 gate** (if necessary): set the `AuditorRegistrySource` to nil in the wire layer. The Build call will log `gossipnet: auditor-scope gate DISABLED ...` and finding-class events will flow as in v1.31.0. **This re-opens the L2 backdoor; only do it if a misconfigured registry is blocking legitimate findings.**

3. **Revert to v1.31.0 binary** if the issue is in the code rather than the data. The pre-v1.32.0 binary cannot consume on-log `AuditorRegistrationV1` entries but ignores them safely (they are admitted as normal entries and walked nowhere).

## Coordination with other repositories

The v1.32.0 adoption in `ledger` does NOT block the rest of the platform from operating, but two downstream adoption PRs are recommended for full benefit:

- **`tooling`** — should adopt the materialized walker views (~480 LOC). Without this, operators need to manually scan the log to verify on-log entries; with this, they get a CLI command.
- **`judicial-network` Phase 3** — should drop `cfg.Witness.Sets`, swap `StaticEndpoints` for `*DefaultAuthoritativeResolver` (~300 LOC). Until JN adopts, its peer-ledger queries will still resolve URLs from JN's own config rather than from the ledger's on-log declarations.

---

## Acceptance criteria for "v1.32.0 fully adopted" on a network

1. `endpoint_source{source="config_canary_fallback"}` is zero across all surfaces.
2. `LEDGER_WITNESS_ENDPOINTS` and `LEDGER_PARENT_ADMISSION_URL` removed from deployment manifests.
3. At least one `AuditorRegistrationV1` per active auditor admitted on-log.
4. At least one `WitnessEndpointDeclarationV1` per witness in the set admitted on-log.
5. `auditor_scope_reject_total{reason="no_registry"}` is zero.
6. `auditor_scope_reject_total{reason="scope_mismatch"}` is zero (no auditor publishing out of scope).
7. CI green on the `ledger` binary version running in production.

When all 7 hold for 7 consecutive days, the network has completed v1.32.0 adoption and the backdoors v1.32.0 was designed to close are operationally closed for that network.
