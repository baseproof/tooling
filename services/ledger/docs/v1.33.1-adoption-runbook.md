# baseproof v1.33.1 SDK Adoption Runbook

Operator-facing rollout, verification, and rollback protocol for the
ledger's adoption of baseproof SDK v1.33.1, succeeding the v1.33.0
rollout described in PR #178.

## Scope of this release

v1.33.1 is a refinement of v1.33.0, NOT a new SDK feature surface.
No new on-log entry kinds, no new env vars, no new endpoints. The
release addresses:

1. **SDK API split.** `network.AuditorRegistration.AuthorizedFor(kind)`
   is REMOVED at v1.33.1 (not deprecated) and replaced by
   `AuthorizedForClaim(kind)` + `AuthorizedForProof(kind)`. The
   ledger's `gossipnet/auditor_scope_gate.go` is the only consumer;
   it now calls `AuthorizedForClaim` exclusively (the gate's
   `isFindingKind` early-exit filters to claim-class only — proof-
   class kinds never reach the authorization branch and are
   intentionally NOT gated on auditor scope; their authority lives
   in the embedded cryptographic proof).

2. **PR #178 Gap 3 reachability fix.** PR #178's `recordCompromise`
   hook was placed AFTER the claim-class early-exit, making it
   unreachable for `KindAuditorCompromise` events. v1.33.1 hoists
   the compromise branch to the top of `Append` so the cutoff is
   recorded on every broadcast. Without this fix, the
   `auditor_scope_reject_total{reason="auditor_compromised"}`
   counter could never increment — the gate's compromise-fast-
   reject path was dead code.

3. **PR #178 Gap 2 gate/API consistency fix.** The materialized
   projection at `GET /v1/network/auditors` previously served the
   raw registration scope while the gate enforced the amendment-
   merged scope. v1.33.1 promotes the fetcher to
   `internal/auditorregistry` and rewires it to call
   `network.ResolveAuditorAt(records, amendments, did, asOf)` per
   DID, locking the gate↔API consistency invariant.

## Rollout protocol

### Pre-deploy verification

```bash
# Confirm the SDK pin
grep "clearcompass-ai/baseproof " go.mod
# Expected: github.com/baseproof/baseproof v1.33.1

# Confirm the gate calls AuthorizedForClaim, not AuthorizedFor
grep -n "AuthorizedFor" gossipnet/auditor_scope_gate.go
# Expected: only AuthorizedForClaim references (one call site + docs)

# Confirm the projection is amendment-aware
grep -n "ResolveAuditorAt" internal/auditorregistry/fetcher.go
# Expected: per-DID call inside LoadCurrentAuditors

# Confirm full test suite is green
go test ./... 2>&1 | grep -E "^(ok|FAIL)" | grep FAIL
# Expected: no output
```

### Deploy sequence

This release is a **single-step binary swap**. No staged rollout
required because:

- No new env vars to introduce
- No new on-log entry kinds to admit
- No new endpoints whose absence would break downstream consumers
- The gate's enforcement semantics are unchanged at the wire level;
  only the SDK method name + the projection's data source change

Steps:

1. Build the v1.33.1 binary.
2. Roll the deployment one node at a time (or per-network policy).
3. Verify the post-deploy checks below at each node.

### Post-deploy verification

1. **Method-rename correctness.** The compile fix is binary —
   either the binary built or it didn't. No runtime check needed.

2. **Compromise tracking is live.** Simulate a compromise broadcast
   in a staging environment using the SDK helper:

   ```go
   f, err := findings.NewAuditorCompromiseFinding(
       "did:web:test-auditor", 100, "rollout verification")
   ```

   POST the event to `/v1/gossip`. After a successful append, query
   the metrics:

   ```
   # If the gate hooks fire correctly, the compromise IS recorded
   # in the in-memory map. There is no metric for "compromise
   # recorded"; the success signal is that a SUBSEQUENT finding
   # from the same DID at asOf >= 100 is rejected with
   # auditor_scope_reject_total{reason="auditor_compromised"}.
   ```

   Issue a follow-up finding from the test DID at asOf > 100; the
   counter MUST increment. If it does not, the recordCompromise
   hook is still dead — investigate `auditor_scope_gate.go`
   immediately.

3. **Projection amendment-merge.** With at least one amendment
   admitted on-log:

   ```bash
   curl -s https://ledger.example.org/v1/network/auditors | jq '.auditors[] | {did: .auditor_did, scope: .scope}'
   ```

   The reported `scope` MUST reflect the most recent amendment's
   NewScope (not the original registration's Scope). To verify:

   - Find a DID with both a registration AND a later amendment on-log
   - Independently compute the expected merged scope using
     `network.ResolveAuditorAt(records, amendments, did, asOf)`
   - The projection's `scope` string MUST match the SDK-computed value

## Observability — counter reasons

`auditor_scope_reject_total{reason, kind}` reasons in scope for this
release (unchanged set, but `auditor_compromised` is now actually
reachable):

| Reason | When it fires |
|---|---|
| `no_registry` | gate has no AuditorRegistrySource wired (fail-closed bootstrap-window posture) |
| `registry_error` | registry walker returned an error |
| `amendment_error` | amendment walker returned an error (v1.33.0+; fail-closed mirror of registry_error) |
| `not_registered` | originator DID has no AuditorRegistrationV1 record at asOf |
| `retired` | most recent registration has RetiredAt ≤ asOf.Sequence |
| `no_registry` | ResolveAuditorAt returned ErrAuditorRecordsEmpty |
| `scope_mismatch` | registered but Scope (after amendment merge) does not cover ev.Kind |
| `auditor_compromised` | originator self-broadcast KindAuditorCompromise; finding asOf.Sequence ≥ compromise cutoff |

Alert thresholds (suggested):

- `auditor_compromised`: any non-zero rate is operator-relevant; investigate the broadcast event and the post-broadcast finding traffic
- `amendment_error`: any non-zero rate indicates the amendment walker is unhealthy; the gate is failing closed for all findings, which is correct behaviour but blocks legitimate auditor traffic
- `scope_mismatch`: low steady rate is acceptable (auditors testing scope boundaries); a sudden spike suggests a misconfigured auditor or an attempted unauthorized broadcast

## Operator playbook — compromise broadcast

When an auditor's key is compromised, the playbook is:

1. Construct a `KindAuditorCompromise` event using the SDK helper:

   ```go
   f, err := findings.NewAuditorCompromiseFinding(
       compromisedDID,
       cutoffSequence,  // the first log sequence at which the key is no longer trusted
       "<incident reason>",
   )
   body, _ := f.EncodeWireBody()
   ```

2. POST the event to the network's gossip ingest. The ledger's gate
   will:
   - Decode the wire body via `findings.AuditorCompromiseFromWire`
   - Insert `(compromisedDID → cutoffSequence)` into the gate's
     in-memory compromise set (lowest-seq-wins on rebroadcast)
   - Persist the event to the underlying gossip Store

3. Every subsequent finding from `compromisedDID` at
   `asOf.Sequence >= cutoffSequence` will be rejected with the
   `auditor_compromised` counter incrementing.

4. The cutoff is forward-only. Findings already admitted at
   `asOf < cutoffSequence` are NOT retroactively invalidated — the
   gate's contract is point-in-time authorization, not after-the-fact
   review.

5. To re-authorize the auditor under a NEW key, the network publishes
   a fresh `AuditorRegistrationV1` entry under the new key. The
   gate's compromise set tracks DIDs not keys — re-registration
   under the same DID is permitted but the operator should confirm
   the compromised key has been rotated out of the network's
   SignaturePolicy before doing so.

## Rollback procedure

The release is fully wire-compatible with v1.33.0:

- No on-log entry kinds were introduced or changed
- No counter names were introduced or changed
- No endpoint paths were introduced or changed
- The gate's wire-level enforcement semantics are unchanged (the
  changes are internal: method-rename + reachability fix + projection
  data-source fix)

To roll back, redeploy the previous v1.33.0 binary. No on-log state
needs reverting; no operator config needs adjusting. The compromise
set is in-memory and rebuilds on next gossip iteration (or stays
empty until the next compromise broadcast under the rolled-back
binary, which is the pre-v1.33.1 behaviour).

The known regression on rollback: the projection at
`/v1/network/auditors` will revert to serving the raw registration
scope, and the `recordCompromise` hook will go back to being
unreachable. Both are existing pre-v1.33.1 conditions and do not
represent new operator risk — they are the conditions this release
fixed.

## Acceptance criteria

Per network, this release is "fully adopted" when:

1. The v1.33.1 binary is running on every node.
2. `go test ./... ` is green on the running ledger source.
3. At least one `KindAuditorCompromise` event has been broadcast and
   verified through the gate (staging is sufficient — production-
   environment broadcasts are by definition rare).
4. The `/v1/network/auditors` projection has been spot-checked
   against an SDK-side `ResolveAuditorAt` recomputation for a DID
   with at least one amendment on-log (or the network has no
   amendments yet, in which case the projection matches the
   registration scope by definition).
5. The `auditor_scope_reject_total{reason="auditor_compromised"}`
   counter is exported and observable; if no compromise broadcasts
   have occurred in production, the steady-state value is correctly
   zero.

When all five hold for 7 consecutive days, v1.33.1 is operationally
adopted for that network.
